## What the next session should know

### 1. Fuel model — sliding both ways now (v4.5.11/v4.5.12)

Anthropic moved the weekly limit from fixed Thursday-21:00-CET to per-user
sliding 7-day windows. Session was already sliding via `reset_in_minutes`.

Implementation:

- `FuelConfig.weeklyResetTime` (ISO) is the user-supplied next reset. When
  set and in the future, IS the next reset.
- `weeklyResetDay/Hour` are FALLBACK only (used pre-first-report or after
  the recorded reset has passed). They're auto-derived from `weeklyResetTime`
  on every save in the user's timezone, so the cycle continues correctly.
- `nexus_log_usage` accepts `weekly_reset_in_hours` (relative) or
  `weekly_reset_at` (ISO). `POST /api/usage` and `localApi.ts` both honor.
- `getNextWeeklyReset(config)` prefers `weeklyResetTime`; falls back to
  day-of-week + hour.
- `buildTimingInfo()` derives the `resetsAt` label from the actual next-reset
  Date and adds `source: 'reported' | 'estimated'`.
- `clock.ts` calendar week-strip derives `isWeeklyReset` from
  `nextWeeklyReset.getDay()` (was hardcoded Thursday).
- `DEFAULT_CONFIG` flipped to Sat/10 (was Thu/21) for first-run.

CC memory `usage_schedule.md` rewritten — explicit "no more Thursday" note,
new MCP call shape with `weekly_reset_in_hours` documented.

## New patterns from v4.5.9 → v4.5.12

**Sparkline component** — 56×14 SVG polyline with last-day dot. Stable
deterministic shape from a fixed-length count array. See `Fleet.jsx`
`Sparkline`. Pattern reusable for any "trend in N buckets" UI.

**Live-tail indicator** — three-state pulse (live / quiet / stale) re-ticking
every 5s so age drifts visibly without WS hookup through the provider. See
`Log.jsx` `liveState`. Useful anywhere "is this thing fresh?" matters.

**Per-row dots breakdown** — four colored dots sized by count. Compact way to
show breakdowns inline in dense lists. See Centrality `byType` in `Graph.jsx`.

**Auto-derive cycle from one observation** — when storing a sliding-window
target time, also derive the cyclic fallback (day-of-week + hour) so the
cycle continues after expiry. Pattern in `usage.ts` for weekly reset; could
apply to any periodic-but-user-reported schedule.

**Single source of truth for shared copy** — `client/src/lib/fuelLabels.js`.
Same pattern would apply if other strings start drifting between modules.

**Display-time data sanitize** — for legacy bad data that we can't (or
shouldn't) rewrite in place, scrub at render time. See `Log.jsx`
`sanitizeMessage` for the `(_project)` cleanup. Store stays immutable; UI
never shows the bug.
